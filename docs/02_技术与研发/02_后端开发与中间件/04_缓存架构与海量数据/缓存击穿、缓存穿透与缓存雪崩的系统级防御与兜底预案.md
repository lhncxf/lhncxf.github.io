# 💊 疑难踩坑与排错备忘：缓存击穿、缓存穿透与缓存雪崩的系统级防御与兜底预案

> **使用场景**：记录在全栈开发（Node.js BFF / 后端聚合层）中，高并发场景下 Redis 缓存架构最致命的“三座大山”引发的宕机血案。从现象复盘到根因剖析，再到企业级的防御基建闭环。

## 📌 一、 故障复盘 (The Incident)
*那些年，我们把数据库打挂的瞬间。*

*   **现象 1（电商大促的午夜）**：某首页千万级曝光的“爆款秒杀商品”由于 Redis 缓存刚好到期（TTL 结束），就在这一秒内，瞬间涌入了 5 万个并发请求。因为 Redis 里没数据，这 5 万个请求全部毫无防备地砸向了后端的 MySQL。MySQL 的 CPU 瞬间 100%，连接池被打满，整个网站瘫痪 10 分钟。**这是典型的【缓存击穿】。**
*   **现象 2（凌晨的黑客狂欢）**：监控显示，有大量请求在疯狂请求一个 ID 为 `-1` 或极其离谱的商品详情页。这个商品在 Redis 里显然没有，于是每次请求都去查 MySQL。MySQL 查不到，返回空，导致 Redis 始终不会缓存这个结果。黑客利用这个漏洞，每秒发动万级请求，MySQL 再次阵亡。**这是典型的【缓存穿透】。**
*   **现象 3（机房断电或批量过期）**：因为我们在半夜 12 点跑了个脚本，把几百万个热点 Key 设置了相同的过期时间（都是 2 小时后过期）。结果在凌晨 2 点那一刻，海量缓存同时失效，或者由于网络波动 Redis 整个集群短暂不可用。此时所有的业务流量全部回源到 MySQL，数据库如同雪崩般被压垮。**这是典型的【缓存雪崩】。**

## 💡 二、 根因分析与排错链路 (Root Cause Analysis)
*为什么加了缓存，反而死得更惨？因为你只懂 GET 和 SET，不懂防御。*

### 2.1 缓存击穿的根因：热点 Key 过期的瞬间真空期
*   **原因**：一个被极高并发访问的 Key 突然过期了。在第一个请求去 MySQL 查数据并写回 Redis 的这几十毫秒的真空期内，后面的几万个请求已经到了，它们同样发现缓存没命中，于是集体去查 MySQL。
*   **企业级解法 (Mutex 互斥锁)**：
    *   在 Node.js 中，发现缓存没命中，不是直接去查 MySQL，而是先尝试用 Redis 的 `SETNX`（Set if Not eXists）去抢一把分布式锁。
    *   抢到锁的那个请求，去查 MySQL，然后把数据写回 Redis，最后释放锁。
    *   没抢到锁的请求，让它们 `await sleep(50ms)` 睡一会儿，醒来后再去查 Redis，这时候数据已经在了。

### 2.2 缓存穿透的根因：查一个根本不存在的数据
*   **原因**：业务逻辑是“缓存查不到 -> 查 DB -> DB查不到 -> 结束”。黑客只要捏造不存在的 ID 狂刷，就能绕过缓存直接打 DB。
*   **企业级解法 (布隆过滤器 Bloom Filter + 缓存空对象)**：
    *   **缓存空对象**：如果 DB 查不到，我们依然把这个 ID 存进 Redis，值设置为空字符串 `""`，过期时间设短一点（比如 5 分钟）。这样接下来的恶意请求就会被 Redis 的 `""` 挡住。
    *   **终极杀招（布隆过滤器）**：把所有合法商品的 ID 提前塞进布隆过滤器。当请求来时，先过一遍布隆过滤器：它说“绝对没有”，那就一定没有，直接返回 404，连 Redis 都不用查；它说“可能有”，再去查 Redis 和 DB。

### 2.3 缓存雪崩的根因：同一时刻大面积过期 / Redis 宕机
*   **原因**：Key 的 TTL 设得太规整，或者 Redis 单点故障。
*   **企业级解法 (打散过期时间 + 多级缓存 + 限流降级)**：
    *   **打散 TTL**：在给 Key 设置过期时间时，加上一个 1-5 分钟的随机值 `TTL + Math.random() * 300`，让它们别在同一秒集体自杀。
    *   **高可用集群与限流**：把单机 Redis 升级为 Redis Cluster（主从哨兵模式）。同时在 Node.js (BFF层) 加上限流器（如 Token Bucket 算法），如果一秒内涌入 10 万并发，直接砍掉 8 万，给他们返回“系统繁忙请稍后重试”，死保 MySQL 活下来。

## ⚙️ 三、 终极方案架构设计 (Ultimate Solution & Architecture)

```javascript
// Node.js (BFF) 视角的防击穿/穿透高可用缓存读取骨架
async function getProductDetail(productId) {
  // 1. 布隆过滤器前置拦截（防穿透第一道防线）
  if (!await bloomFilter.mightContain(productId)) {
    return null; // 黑客捏造的非法 ID，直接滚蛋
  }

  // 2. 查 Redis
  const cacheKey = `product:detail:${productId}`;
  let data = await redis.get(cacheKey);

  // 3. 缓存命中，且是防穿透的“空值标记”
  if (data === "NULL_CACHE") return null;
  // 4. 正常命中业务数据
  if (data) return JSON.parse(data);

  // 5. 缓存没命中（可能是热点过期了），准备查 DB
  const lockKey = `lock:product:detail:${productId}`;
  
  // 抢分布式互斥锁（防击穿核心）
  const isLockAcquired = await redis.setnx(lockKey, '1', 'EX', 5); // 锁 5 秒防死锁
  
  if (isLockAcquired) {
    try {
      // 只有抢到锁的倒霉蛋去查 MySQL
      const dbData = await db.query('SELECT * FROM products WHERE id = ?', [productId]);
      
      if (!dbData) {
        // 查不到数据？缓存空对象，设极短的随机 TTL（防穿透第二道防线）
        await redis.set(cacheKey, "NULL_CACHE", 'EX', 60 + Math.floor(Math.random() * 60));
        return null;
      } else {
        // 查到了？写回缓存，设打散的随机 TTL（防雪崩防线）
        const randomTTL = 3600 + Math.floor(Math.random() * 600); // 1小时 + 0~10分钟随机
        await redis.set(cacheKey, JSON.stringify(dbData), 'EX', randomTTL);
        return dbData;
      }
    } finally {
      // 释放锁
      await redis.del(lockKey);
    }
  } else {
    // 没抢到锁的，等 50 毫秒后再重新去 Redis 查一遍（这时候那个拿锁的应该已经把数据放进去了）
    await sleep(50);
    return getProductDetail(productId); // 递归重试
  }
}
```

## 📝 四、 沉淀与复盘 (Takeaways)
*   作为写 Node.js BFF 的全栈，必须对 MySQL 有绝对的敬畏之心。Redis 可以随便挂（有持久化和主从恢复），但 MySQL 一挂，所有的写操作（订单、支付）全停，这是 P0 级事故。
*   缓存的三大问题本质上是：**穿透**是“无中生有（瞎编ID）”；**击穿**是“单点突破（爆款过期）”；**雪崩**是“全面崩溃（集体自杀）”。防御的核心思想就是：挡住恶意的、排队限流正常的、打散过期的。
