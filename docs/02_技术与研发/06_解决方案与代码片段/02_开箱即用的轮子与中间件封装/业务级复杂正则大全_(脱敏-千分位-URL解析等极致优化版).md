# 业务级复杂正则大全 (脱敏-千分位-URL解析等极致优化版)

> **使用场景**：沉淀业务开发中高频出现的复杂正则方案。拒绝由于“回溯（Backtracking）”导致的性能灾难，提供生产环境验证过的极致优化版本。

## 1. 痛点与需求场景 (Context)

在长达十年的前端摸爬滚打中，我见过太多因为一个“贪婪正则”导致浏览器标签页假死甚至服务端 CPU 飙升的惨案。
* **原始痛点**：
  * **回溯陷阱**：简单的 `(a+)+b` 匹配 `aaaa...c` 会导致指数级回溯。
  * **业务复杂性**：比如 URL 解析不仅要快，还要能精准拆分协议、域名、路径及参数。
  * **数据安全**：在日志打印前，必须对手机号、身份证、姓名等进行脱敏，且不能破坏原始字符串结构。
* **预期目标**：
  * 性能：在万量级字符下依然保持毫秒级响应。
  * 准确性：覆盖业务边缘情况（如带小数的千分位、IPv6 地址等）。

## 2. 核心架构与设计思路 (Design & Best Practice)

* **拒绝贪婪，拥抱非贪婪/独占**：尽可能缩小匹配范围，利用 `*?` 或 `+?` 减少回溯，或者在支持的环境下使用独占模式。
* **先排除后匹配**：利用 `[^...]` 排除字符集比 `.` 匹配更快。
* **捕获组最小化**：如果不需提取，使用 `(?:...)` 非捕获组，节省内存消耗和计算资源。
* **零宽断言的妙用**：在千分位和脱敏场景中，利用 `(?=...)` 和 `(?<=...)`（注意 Safari 兼容性）实现不占位的精准定位。

## 3. 开箱即用：核心代码骨架 (Implementation)

### 3.1 手机号/身份证脱敏（不伤性能的替换）
```javascript
/**
 * 手机号脱敏：13812345678 -> 138****5678
 * 优化点：直接通过索引捕获，避免全量扫描
 */
const maskPhone = (phone) => phone.replace(/^(\d{3})\d{4}(\d{4})$/, '$1****$2');

/**
 * 身份证脱敏：320101199001011234 -> 3201**********1234
 */
const maskIDCard = (id) => id.replace(/^(\d{4})\d+(\d{4})$/, (match, p1, p2) => {
  return p1 + '*'.repeat(match.length - 8) + p2;
});
```

### 3.2 极致性能：数字千分位（零回溯版）
```javascript
/**
 * 经典方案：(?=(\d{3})+(?!\d))
 * 优化思路：处理带小数点的情况，且保证在大数字下的性能
 */
const formatThousand = (num) => {
  const parts = num.toString().split('.');
  // 针对整数部分进行正则匹配
  parts[0] = parts[0].replace(/\B(?=(\d{3})+(?!\d))/g, ',');
  return parts.join('.');
};
```

### 3.3 工业级 URL 解析（非贪婪拆解）
```javascript
/**
 * 解析 URL 各个组成部分
 * 采用非贪婪模式 + 非捕获组
 */
const urlRegex = /^(?:([^:/?#]+):)?(?:\/\/([^/?#]*))?([^?#]*)(?:\?([^#]*))?(?:#(.*))?/;
const parseUrl = (url) => {
  const matches = url.match(urlRegex);
  return {
    protocol: matches[1],
    host:     matches[2],
    path:     matches[3],
    query:    matches[4],
    hash:     matches[5]
  };
};
```


## 4. 边界情况与避坑补充 (Edge Cases & Gotchas)

* **Safari 的 Lookbehind 坑**：尽管现代浏览器大部分支持 `(?<=...)`，但在某些 iOS 版本的老 Safari 中会直接抛出语法错误。建议在 Hybrid H5 开发中慎用，改用传统的捕获组 + 替换逻辑。
* **正则 ReDoS 攻击**：在处理用户输入（如搜索框、过滤器）时，严禁使用嵌套的重复结构（如 `(a+)+`），必须通过前端代码层做长度限制（如 `maxlength`）。
* **全局匹配的 lastIndex**：使用 `g` 标识符的正则对象，在多次 `test()` 调用时会记录位置。老炮的做法是每次使用前重置 `re.lastIndex = 0` 或者干脆直接用字面量。
