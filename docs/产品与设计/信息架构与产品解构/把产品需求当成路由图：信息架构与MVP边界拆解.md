# 把产品需求当成路由图：信息架构与 MVP 边界拆解

> **使用场景**：本文作为从 10 年前端专家转型产品设计的认知重塑。将前端工程中的“路由管理”与“组件化思维”映射到产品的信息架构（IA）与用户故事地图中，通过工程视角重新审视 MVP 的交付边界。

## 1. 探究动机 (Why Now?)
在做了 10 年前端开发后，我发现很多时候项目的延期或失败，并不是技术实现太难，而是“需求路由”根本没跑通。产品经理给出的原型图往往是散点式的，缺乏严密的逻辑闭环。
* **现状盲区**：过去我只关注 `App.vue` 下的路由嵌套和状态流转，认为产品设计就是画几张 UI 稿。但实际接触产品规划后，我发现如果把产品看作一个复杂的单页应用（SPA），那么信息架构（IA）就是那个最底层的 `router/index.ts`。

## 2. 核心机制解构 (Mental Model)

### 2.1 信息架构（IA）= 路由地图 (Route Mapping)
产品的信息架构决定了用户在产品中的位置感。正如 Vue Router 中的嵌套路由（Nested Routes），一个好的 IA 应该能让用户清晰地知道：
* **一级路由**：Tab 栏或侧边栏，定义了产品的大功能版块（如首页、工作台、设置）。
* **二级/三级路由**：功能内部的跳转逻辑。
* **动态路由**：基于不同权限（RBAC）或数据 ID 渲染的特定页面。

### 2.2 用户故事地图 = 组件树与 Hooks (Component Hierarchy)
User Story Mapping 不是简单的 Feature List。它更像是在规划整个应用的组件树：
* **骨干故事（Backbone）**：App 的主体框架（Layout）。
* **任务流程（User Tasks）**：业务逻辑组件（Smart Components）。
* **细节需求（Details）**：UI 原子组件或自定义 Hooks。

### 2.3 MVP = Unstyled HTML Structure
很多产品容易在 MVP 阶段陷入“细节泥潭”。用前端思维来看，MVP 应该是在写 CSS 之前的那套 HTML 结构：
* **核心交付**：先跑通数据流（Action -> Store -> View）。
* **边界拆解**：MVP 只负责渲染正确的 DOM 节点和基础逻辑。至于“动画特效”、“深色模式”、“极致的响应式适配”，那是 V1.1 甚至 V2.0 后的 `style.css` 任务。

```typescript
// 用最简化的路由逻辑还原产品 IA
const productRouter = [
  {
    path: '/core-service',
    component: MVP_Boundary, // MVP 阶段只保证核心业务组件可用
    children: [
      { path: 'must-have', priority: 'high' },
      { path: 'nice-to-have', priority: 'low' } // MVP 阶段此分支直接 return null
    ]
  }
];
```

## 3. 认知反转与横向对比 (Mental Shift & Comparison)
* **认知刷新**：以前觉得 MVP 是“简陋版”，现在意识到 MVP 是“骨架版”。如果一个产品在去掉所有 CSS 后无法通过 HTML 结构完成业务闭环，那它的逻辑底层（IA）一定有问题。
* **横向对比**：传统瀑布流开发像是写原生 JS，每一步都必须完整；而现代产品设计更像 React 的并发模式（Concurrent Mode），IA 确定后，功能模块可以按优先级异步渲染、逐步上线。

## 4. 业务投影与延伸思考 (Extension)
* **业务指导 1**：在评审需求文档（PRD）时，我会下意识地先画出它的“路由嵌套图”。如果发现某个功能跳转需要超过 3 层深度且无法通过面包屑返回，说明 IA 冗余，需要重构。
* **业务指导 2**：定义 MVP 边界时，我会问自己：“这个功能如果我不写任何样式代码，用户还能完成核心操作吗？”。如果不能，它就是 Must-have；如果能，它大概率可以推迟到下一迭代。
* **延伸探索**：接下来需要补齐的是“竞品拆解”的方法论。如何像分析竞品源码（F12 查看网络请求和 DOM 结构）一样，去反推竞品背后的产品逻辑与商业闭环。
