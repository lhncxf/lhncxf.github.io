# 💊 前端内存泄漏与全链路性能监控体系建设指南

> **使用场景**：在项目上线后，面对首屏慢、运行卡顿、甚至页面崩溃（Out Of Memory）等疑难杂症时的排查指南。涵盖从 Performance API 到 V8 内存快照分析，再到搭建 Sentry/Datadog 等全链路监控底座。

## 📌 一、 报错现象与复盘 (The Incident)
*   **现象描述**：某内部高频使用的千万级数据大盘页面，用户反馈打开后放置不管，半小时后浏览器标签页直接卡死崩溃。
*   **初步定位**：使用 Chrome DevTools 的 `Task Manager` 观察，发现页面在空闲状态下，JS 堆内存从初始的 50MB 一路飙升到 1.5GB 直至崩溃（OOM）。
*   **核心原因**：典型的闭包内存泄漏。WebSocket 推送高频数据时，未清理过期的高阶函数闭包；以及 Vue 组件销毁时，全局 `window.addEventListener` 和 `echarts.init` 实例未手动 `dispose`。

## 💡 二、 根因分析与排错链路 (Root Cause Analysis)
*排查内存泄漏的唯一真理是：让数据说话，而不是看代码瞎猜。*

### 2.1 捕获罪魁祸首：V8 内存快照分析 (Heap Snapshot)
1.  **录制快照**：打开 DevTools -> Memory -> Heap Snapshot。分别在“刚进入页面”、“操作一段时间后”、“离开页面后”录制三次快照。
2.  **比对差异 (Comparison)**：将视图切换为 `Comparison` 模式，比对第三次和第一次的差异。重点关注 **Detached DOM trees**（脱离文档流但仍被 JS 强引用的 DOM 节点）和被标黄/标红的 `Closure` 对象。
3.  **顺藤摸瓜 (Retainers)**：点击泄漏的对象，查看底部的 `Retainers` 树，这棵树会明确告诉你：究竟是哪一行的哪个变量（比如一个被误设为全局变量的 Array）死死拽着它不放，导致 GC（垃圾回收器）无法回收。

### 2.2 防患未然：从弱引用到规范清理
*   **WeakMap / WeakSet**：在封装底层框架或缓存大型数据对象时，全面改用 WeakMap。它的键是弱引用，当 DOM 被销毁时，WeakMap 里的数据会随着下一次 GC 自动蒸发，无需手动清理。
*   **生命周期成对出现**：立下军令状，凡是在 `mounted` / `useEffect` 里执行的 `setInterval`、`addEventListener`、第三方库实例化（ECharts、Mapbox），必须要在 `unmounted` / `return () => {}` 里对应执行 `clearInterval`、`removeEventListener` 和 `dispose()`。

## ⚙️ 三、 终极解法与监控基建 (Solutions & Infrastructure)
*靠人肉排查是不行的，高级前端需要的是“哨兵”。*

### 3.1 全链路性能监控指标 (Web Vitals)
不再看毫无意义的 `DOMContentLoaded`，全面拥抱 Google 的 **Web Vitals** 核心指标：
*   **LCP (Largest Contentful Paint)**：最大内容绘制。真实反映首屏什么时候加载完（比如首屏最大的头图出来了）。目标：< 2.5s。
*   **FID (First Input Delay) / INP (Interaction to Next Paint)**：首次输入延迟。反映用户点一个按钮，页面多久才给出反应。目标：< 100ms。
*   **CLS (Cumulative Layout Shift)**：累积布局偏移。反映页面加载时会不会突然跳动（比如一张图片突然撑开把文字挤下去）。目标：< 0.1。

### 3.2 埋点与监控平台搭建 (Sentry / Datadog)
利用浏览器底层的 `PerformanceObserver` API，在全局无侵入地拦截上述性能指标，以及未捕获的 Error 和 Promise Rejection，定时通过 `sendBeacon` API（它可以在浏览器关闭时依然保证数据发出）上报给自建的日志聚合平台。
结合 `source-map`，让线上的乱码报错直接映射到你 Git 仓库的具体代码行。

## 📝 四、 避坑与延伸 (Pitfalls & Extension)
*   **避坑：console.log 也是泄露源**：你没听错，在线上环境打印复杂的大型 Object 时，为了保证你在打开开发者工具时还能点开查看对象内部，浏览器会对这个 Object 保持强引用。生产环境的构建必须配置 `terser` 或 `swc` 剔除掉所有的 console。
*   **反思：SPA 的天然劣势**：单页应用（SPA）不刷新页面的特性，注定了它是内存泄漏的重灾区（传统多页应用跳一次链接，上个页面的内存直接清空）。这意味着我们在写 SPA 时，对内存的敬畏之心必须比写传统网页高出十倍。
