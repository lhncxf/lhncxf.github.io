# 电商底层抽象：“人、货、场”模型的前端工程化映射

在电商领域，“人、货、场”是绕不开的经典模型。作为一名干了十年的前端，我习惯把复杂的业务逻辑拆解成可维护的代码架构。这篇文章不谈虚头巴脑的商业逻辑，咱们从工程实现的角度，看看如何把这三根支柱映射到前端的组件树、数据流和基础设施上。

## 1. 人（People/User）：身份识别与权限状态机

在前端工程中，“人”的本质是**状态（State）**与**凭证（Identity）**。

### 跨端身份映射
现在的电商早已不是单兵作战，微信小程序、支付宝、App、H5 全都要。
- **UnionID 与 OpenID**：在业务逻辑层，我们要建立一套身份转化矩阵。前端 SDK 需要在 `App.onLaunch` 时，自动探测当前环境并获取对应的身份标识。通过 `UnionID` 实现多端唯一性锚定，而 `OpenID` 则作为特定平台的通信句柄。
- **OAuth 2.0 与 SSO**：实现一套标准的单点登录流（Flow）。在 `axios` 拦截器里，通过 `refresh_token` 机制实现无感刷新。这就像是前端的一个后台常驻进程，确保用户的身份始终处于 `Authenticated` 状态。

### JWT Payload 优化
别把所有用户信息都塞进 Token。合理的做法是：
- **Header/Payload**：只保留 `uid` 和 `exp`（过期时间）。
- **权限模型**：对于 B 端（商家后台），采用 **RBAC**（基于角色的访问控制）或 **ABAC**（基于属性的访问控制）。在路由层面，通过 `meta` 字段定义权限位；在 UI 层面，封装一个权限指令（如 `v-permission`），本质上是对 DOM 节点的条件挂载。

### 行为追踪（埋点 SDK）
“人”的行为是业务增长的燃料。
- **声明式埋点**：不要在代码里到处写 `trackEvent`。封装一个埋点组件或指令，通过 `data-track` 属性声明式地收集点击、曝光。
- **队列管理**：埋点数据不能实时发，会拖慢主线程。需要实现一个内部队列，利用 `requestIdleCallback` 或 `Navigator.sendBeacon` 在浏览器空闲时批量上报。

## 2. 货（Goods/Product）：多维状态的 UI 映射

“货”在前端的表现是高度结构化的**数据模型**。

### SPU 与 SKU 的状态组合
一个商品的详情页，本质上是一个复杂的**状态机**。
- **规格矩阵**：前端需要根据后端下发的 SKU 列表，实时计算出各种规格（颜色、尺码、套餐）的交集。这里常用的算法是**无向图**或**幂集算法**，用来判断当前选中的规格组合是否还有库存，从而动态 `disabled` 掉不可选的按钮。
- **实时库存同步**：秒杀场景下，前端的库存数是靠 `WebSocket` 或 `SSE` 推送更新的。这不仅仅是一个数字，它涉及到前端组件的局部刷新逻辑。

### 物理 vs 虚拟：UI 逻辑分支
- **物理商品**：关注物流追踪组件、收货地址校验逻辑。
- **虚拟商品**：关注兑换码生成、账号绑定校验。
在组件设计上，这应该是一套**策略模式**。根据 `goods_type` 动态渲染不同的子组件，而不是写一堆 `if-else`。

## 3. 场（Scene/Traffic）：全链路渲染与动态生成

“场”是用户接触商品的媒介，在技术层面就是**渲染引擎**。

### 多端渲染架构
流量碎片的时代，前端必须具备全栈渲染能力。
- **跨端框架选型**：React Native 解决 App 的性能瓶颈，Taro 或 UniApp 解决小程序的快速铺货。核心思路是“一套代码，多端适配”，通过 `TARO_ENV` 等编译宏来实现特定端的差异化逻辑。
- **SEO 友好型 SSR**：对于公域流量（如搜索引擎、社交分享），必须上 `Next.js` 或 `Nuxt.js`。服务端渲染不仅是为了 SEO，更是为了提升 LCP（最大内容渲染）性能，降低用户在首屏加载时的流失率。

### 灵动页与 Server-Driven UI
促销活动瞬息万变，靠发版解决不了问题。
- **低代码搭建系统**：将 UI 拆解成原子组件（Banner、瀑布流、倒计时）。后端返回一套 JSON Schema，前端通过一个 `DynamicRenderer` 动态解析并渲染。
- **Server-Driven UI**：让后端控制组件的顺序和样式。前端只是一层薄薄的渲染壳，所有的业务逻辑和页面布局都由服务端下发。这种架构极大提升了运营的灵活性，真正做到了“场”的动态化。

## 总结

在资深前端眼中，电商的“人、货、场”绝非商业课本上的名词，它们分别是**身份权限系统**、**复杂状态矩阵**和**动态渲染引擎**。

一个优秀的电商前端工程，应该能像乐高一样：
1. **身份（人）**：隔离在全局状态管理中，随调随用。
2. **商品（货）**：封装成高内聚的组件库，支持复杂的规格交互。
3. **场景（场）**：具备跨平台交付和动态配置能力。

看透了这层映射，剩下的不过是代码的填空题。
