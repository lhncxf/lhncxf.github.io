# 大前端复杂业务场景下的设计模式演进与落地

在前端工程化演进的这十年里，我们早已脱离了单纯的页面堆砌。面对日益增长的业务复杂度，GoF 设计模式不再是 Java 程序员的专利，而是解决大前端架构逻辑解耦、状态管理和组件复用的核心利器。

这篇文章不谈类图，不谈抽象概念，只谈在 React/Vue/Node.js 真实业务中，我们如何通过模式重构代码。

## 1. 策略模式 (Strategy)：终结表单校验与复杂分支的“屎山”

最常见的反模式是在 `submit` 函数里写几十个 `if-else`。策略模式的核心在于将“校验算法”与“业务逻辑”分离。

### 业务场景：动态表单校验
在 B 端复杂的后台管理系统中，一个表单可能根据不同的用户角色、地区有完全不同的校验规则。

```typescript
// 校验策略定义
const validationStrategies = {
  isNonEmpty: (value: string) => value.length !== 0 || '内容不能为空',
  isEmail: (value: string) => /^\w+@\w+\.com$/.test(value) || '邮箱格式不正确',
  minLength: (value: string, length: number) => value.length >= length || `长度不能小于${length}`
};

// 策略执行器
class Validator {
  private cache: any[] = [];
  
  add(value: any, rule: string, ...args: any[]) {
    this.cache.push(() => validationStrategies[rule](value, ...args));
  }

  check() {
    for (let validateFn of this.cache) {
      const msg = validateFn();
      if (typeof msg === 'string') return msg;
    }
  }
}

// 业务调用：完全解耦的线性逻辑
const validator = new Validator();
validator.add(formData.username, 'isNonEmpty');
validator.add(formData.email, 'isEmail');
const error = validator.check();
```

## 2. 发布-订阅模式 (Observer/Pub-Sub)：跨层级通信的血脉

在微前端或大型单页应用中，组件层级可能深不可测。Props Drilling 是地狱，而 EventBus 或状态库（如 Redux/Pinia）的底层正是观察者模式。

### 业务场景：全局事件总线与状态变更
当我们需要在“侧边栏”收起时，让全局的“大屏看板”重绘。

```typescript
// 一个典型的基于 Mitt 或原生实现的 EventBus
class EventEmitter {
  private events: Record<string, Function[]> = {};

  on(type: string, handler: Function) {
    if (!this.events[type]) this.events[type] = [];
    this.events[type].push(handler);
  }

  emit(type: string, ...args: any[]) {
    this.events[type]?.forEach(handler => handler(...args));
  }
}

export const bus = new EventEmitter();

// 侧边栏组件
bus.emit('sidebar-toggle', { collapsed: true });

// 看板组件
bus.on('sidebar-toggle', ({ collapsed }) => {
  resizeCharts();
});
```

## 3. 工厂模式 (Factory)：复杂组件渲染的灵感源泉

前端的工厂模式通常体现为“配置驱动开发”。通过一套配置 JSON，动态生成对应的 UI 组件。

### 业务场景：配置化 CMS 页面渲染
假设你正在开发一个低代码平台，后台返回一堆 JSON 描述页面布局。

```tsx
const ComponentFactory = (config: { type: string, props: any }) => {
  const componentMap = {
    'button': PrimaryButton,
    'input': SearchInput,
    'chart': LineChart,
  };

  const Component = componentMap[config.type] || DefaultPlaceholder;
  return <Component {...config.props} />;
};

// 渲染循环
{pageConfigs.map(config => <ComponentFactory key={config.id} config={config} />)}
```

## 4. 装饰器模式 (Decorator)：优雅的横切关注点处理

在 ESNext 和 TypeScript 中，装饰器让逻辑增强变得极其丝滑。它能在不改变原代码逻辑的前提下，注入权限校验、埋点监控或响应式追踪。

### 业务场景：HOC 与 MobX 响应式增强
高阶组件 (HOC) 本质上就是装饰器模式在 React 中的变体。

```tsx
// 埋点统计装饰器（HOC 形式）
function withLog(WrappedComponent) {
  return class extends React.Component {
    componentDidMount() {
      console.log(`页面渲染耗时统计: ${performance.now()}`);
    }
    render() {
      return <WrappedComponent {...this.props} />;
    }
  }
}

@withLog
class Dashboard extends React.Component { ... }

// 或者 MobX 中的响应式装饰器
class Store {
  @observable count = 0;
  @action increment() { this.count++; }
}
```

## 5. 依赖注入与提供者模式 (DI/Provider)：解决上下文耦合

在大型应用中，我们不希望组件直接依赖某个具体的 API 实现。通过 Provider，我们可以将具体的 Service 注入到整棵组件树中。

### 业务场景：多端适配与主题系统
通过 React Context 实现的依赖注入。

```tsx
// 1. 定义 Service 抽象（接口思想）
const ApiServiceContext = React.createContext<IApiService>(null);

// 2. 注入具体实现（PC 端实现）
const App = () => (
  <ApiServiceContext.Provider value={new PCApiService()}>
    <SubComponent />
  </ApiServiceContext.Provider>
);

// 3. 消费依赖
const SubComponent = () => {
  const api = useContext(ApiServiceContext);
  // 这里不需要关心 api 是来自 PC 还是移动端，只管调用方法
  api.getData().then(res => { ... });
};
```

## 总结：模式不是束缚，而是共识

作为 Senior 开发者，我们要意识到设计模式并不是要增加代码的复杂性。相反，它们是为了**提高代码的预测性**。

当团队成员看到 `Strategy` 文件夹时，他们就知道这里存放的是可扩展的算法；看到 `Provider` 时，就知道这里处理的是全局状态或依赖。这种“共识”才是大前端项目在高速迭代中不至于崩塌的关键。
