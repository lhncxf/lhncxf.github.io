# 单点实战与最佳实践 [千万级数据的表结构拆分（分库分表）与外键约束取舍]

> **使用场景**：随着业务的发展，单表数据量突破 1000 万，MySQL 的 B+ 树索引高度增加，甚至单次 `COUNT(*)` 和复杂 JOIN 都会引发慢查询（几十秒）。系统架构必须向分库分表（Sharding）演进，并在海量并发下重新审视传统关系型数据库的外键（Foreign Key）约束策略。

## 1. 痛点与需求场景 (Context)
* **原始痛点**：
  - **B+树膨胀与慢查询**：单表体积超过 50GB 或行数达到两千万级别时，哪怕索引建得很完美，一次简单的条件筛选可能都要超过 1 秒。如果有频繁的更新或插入，由于锁竞争，数据库性能会悬崖式暴跌。
  - **物理资源瓶颈**：单台服务器的磁盘 IO、CPU 计算能力、连接数是有限的，全靠“加配置（Scale Up）”总有到顶的一天。
  - **外键带来的死锁灾难**：在并发插入/删除主表和子表时，MySQL 需要检查外键完整性，这会产生大量的共享锁（S锁）和间隙锁。如果是高频秒杀、抢购场景，外键大概率会导致大规模的死锁等待，拖垮整个服务。
* **预期目标**：
  - **垂直拆分 (Vertical Partitioning)**：把宽表里不常用的长文本字段（如大段的富文本描述、JSON 配置）剥离出去。
  - **水平拆分 (Horizontal Partitioning/Sharding)**：将 1 亿条用户数据，均匀打散到 100 张表里（`user_00` 到 `user_99`），解决单表体积问题。
  - **应用层级联替代物理外键**：果断抛弃 MySQL 的外键约束。改用代码在应用层保证数据的一致性，拥抱“最终一致性（Eventual Consistency）”。

## 2. 核心架构与设计思路 (Design & Best Practice)
* **水平拆分（Sharding）的路由算法（Sharding Key 的选择）**：
  - **哈希取模 (Hash Modulo)**：最常用的拆分法。比如按 `user_id % 100`，算出该用户的数据落在 `user_00` 还是 `user_99`。**优点**：数据极其均匀；**缺点**：如果未来 100 张表不够了想扩容到 200 张，必须做天翻地覆的重新取模和数据大洗牌（Rehash 数据迁移）。
  - **范围分片 (Range Sharding)**：比如按创建时间，2023年的数据存 `orders_2023`，2024年的存 `orders_2024`。或者 `ID: 0~1000万` 存 db1，`1000万~2000万` 存 db2。**优点**：扩容极简，加机器就行；**缺点**：热点数据全在最新的一台机器上（请求倾斜），老机器闲置。
  - **最佳实践：Hash + Range 结合**：根据 `user_id % 10` 拆库，每个库里再根据月份建表。
* **为什么大厂绝对禁用物理外键（Foreign Key）？**
  - **性能代价极高**：每次 `INSERT / UPDATE / DELETE` 都要查询父表/子表以验证约束。
  - **锁级联**：并发场景下，删除父表数据会导致全盘的子表被锁定，极易发生死锁（Deadlock）。
  - **分库分表是外键的坟墓**：当你把 `users` 表和 `orders` 表拆到了两台不同的物理服务器上，跨物理库的物理外键根本不成立。此时只能靠应用层的分布式事务（或消息队列）来保证关联。

## 3. 开箱即用：核心代码骨架 (Implementation)

这是一个基于 Node.js BFF 层的简单路由分片中间件（Sharding Proxy）。虽然大厂一般使用 Sharding-JDBC (Java) 等专门的中间件，但在 Node.js 轻量级微服务中，我们也需要了解如何利用代码自己算路由。

### 3.1 垂直拆分示例（伪代码思维）

当你有一个超大体积的文章表（包含了 `id, title, author, content`）。其中 `content` 是几兆的富文本，但我们在列表页只查标题，不需要正文。

```sql
-- 原始表（性能灾难）
CREATE TABLE articles (
  id INT PRIMARY KEY,
  title VARCHAR(255),
  author_id INT,
  content LONGTEXT -- 这个巨无霸字段会导致 B+ 树的节点体积变大，降低 Buffer Pool 的缓存效率
);

-- 【最佳实践】重构为垂直拆分：列表读主表，详情查扩展表
CREATE TABLE article_infos (
  id INT PRIMARY KEY,
  title VARCHAR(255),
  author_id INT
);

CREATE TABLE article_contents (
  article_id INT PRIMARY KEY, -- 直接使用业务主键，避免自增
  content LONGTEXT
);
```

### 3.2 强行拔掉外键：在代码层保证级联一致性

既然我们不能使用 `ON DELETE CASCADE`，那业务中删除了用户，怎么同时删除他的订单和日志？
> 答案：使用本地事务（同库），或者基于消息队列（跨库）的柔性事务。

```javascript
// Node.js 代码层模拟外键级联删除 (同库事务)
const deleteUserAndOrders = async (userId) => {
  const connection = await db.getConnection();
  await connection.beginTransaction();

  try {
    // 1. 先删子表（应用层级联）
    await connection.query('DELETE FROM orders WHERE user_id = ?', [userId]);
    await connection.query('DELETE FROM logs WHERE user_id = ?', [userId]);
    
    // 2. 最后删主表
    const [result] = await connection.query('DELETE FROM users WHERE id = ?', [userId]);
    
    await connection.commit();
    return result.affectedRows;
  } catch (err) {
    // 任何一步失败，全盘回滚，保证一致性
    await connection.rollback();
    throw err;
  } finally {
    connection.release();
  }
};
```

### 3.3 简易水平分库路由函数 (Hash 算法)

```javascript
// 假设我们有 4 台数据库实例，每台实例下面有 16 张订单表（总共拆了 64 张表）
const TOTAL_DBS = 4;
const TABLES_PER_DB = 16;
const TOTAL_TABLES = TOTAL_DBS * TABLES_PER_DB; // 64

/**
 * 根据用户 ID，计算他应该落在哪台物理机，以及具体哪张表
 * @param {number} userId 
 */
function calculateShardingRoute(userId) {
  // 1. 算出落在 0 ~ 63 的哪一张虚拟表
  const tableIndex = userId % TOTAL_TABLES; 
  
  // 2. 算出所在的库（每16张表在一个库）: Math.floor(tableIndex / 16)
  const dbIndex = Math.floor(tableIndex / TABLES_PER_DB);
  
  // 3. 算出在该库里的实际表后缀（0~15）: tableIndex % 16
  const actualTableSuffix = tableIndex % TABLES_PER_DB;

  return {
    database: `db_cluster_${dbIndex}`,       // e.g., db_cluster_1
    tableName: `orders_${actualTableSuffix}` // e.g., orders_15
  };
}

// console.log(calculateShardingRoute(10086));
// { database: 'db_cluster_2', tableName: 'orders_6' }
```

## 4. 边界情况与避坑补充 (Edge Cases & Gotchas)
* **分库分表后的分页查询黑洞（Order By / Limit 灾难）**：假设你拆了 10 张表，前端发来请求 `SELECT * FROM orders ORDER BY time DESC LIMIT 1000, 20`。此时中间件必须向 10 张表分别下发 `LIMIT 0, 1020`（总共取出 10200 条数据），在内存里进行合并（Merge）、重新排序，最后再切掉前 1000 条。这会导致应用服务器 OOM（内存溢出）。**终极解法**：绝对禁止深分页。如果有全局搜索、复杂组合条件筛选需求，必须将数据异构同步（如通过 Canal 同步 Binlog）到 Elasticsearch 或 TiDB（分布式列存）中，由搜索引擎负责跨表、跨库的分页统计。
* **跨库 JOIN 的死亡**：分表后，原本 MySQL 一句 `JOIN` 就能搞定的查询直接作废。**解法 1**：在应用层通过两次查询，在内存中拼接（先查主键列表，再用 `IN` 查关联表）。**解法 2**：字段冗余设计。比如订单表中直接存入 `user_name` 和 `user_phone`，不再去关联用户表查名字，用磁盘空间换取查询效率（典型的 NoSQL 化思维）。
