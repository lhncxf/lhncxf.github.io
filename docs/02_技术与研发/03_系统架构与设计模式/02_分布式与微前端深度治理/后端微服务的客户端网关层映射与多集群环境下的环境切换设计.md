# 单点实战与最佳实践 [后端微服务的客户端网关层映射与多集群环境下的环境切换设计]

> **使用场景**：在大型互联网公司（比如阿里、字节或跨境电商）中，后端往往拆分了成百上千个微服务（Java/Go），并且拥有多套独立的部署集群（开发、日常、预发、灰度、生产）。如果前端代码里直接把接口 URL 写死为 `https://user-service.api.com`，在切换测试环境或者某服务下线重构时，前端会被坑死（发几百个版本去改域名）。必须建立一套强健的前端 API 网关（BFF / Nginx / Kong）映射机制和环境变量体系。

## 1. 痛点与需求场景 (Context)
* **原始痛点**：
  - **环境混乱（Env Hell）**：测试同学说“帮我打一个连到日常环境（Daily）的包，因为里面造了数据”，过一会儿产品又说“我要连预发环境（Staging）演示”。前端每次都要去改 `.env` 里的 `VUE_APP_BASE_API` 然后花几分钟构建、发版。
  - **跨域地狱（CORS Nightmare）**：前端跑在 `localhost:3000`，去请求十几个不同域名的后端的接口。后端 Java 开发抱怨“凭什么要我配 CORS？”，于是前端在本地自己配反向代理（Proxy），但一上生产环境就 403 跨域报错。
  - **微服务暴露面太大**：前端直接知道 `user-service`, `order-service`, `pay-service`。一旦后端把 `order-service` 拆成 `cart` 和 `checkout`，前端就要跟着全局搜索替换 API 路径，牵一发而动全身。
* **预期目标**：
  - **统一的同域入口（API Gateway / BFF）**：前端只认识唯一的入口 `https://api.mycorp.com`。不论后面挂了几百个微服务，统统由网关基于路径路由（`/api/user/*` 转发给 Java，`/api/order/*` 转发给 Go）。
  - **零构建切换环境（Dynamic Env）**：前端构建出的是一份纯粹的静态资源（Docker Image），不需要为每个环境打一个包（Build Once, Run Anywhere）。环境配置在 Nginx 或 K8s ConfigMap 中动态注入。

## 2. 核心架构与设计思路 (Design & Best Practice)
* **网关层映射 (API Gateway Routing)**：
  - 核心思想：把跨域问题彻底扼杀在服务端。前端的域名是 `app.mycorp.com`，网关域名可以也是它自己，或者子域名 `api.mycorp.com`。所有的 AJAX 请求全部发给网关（携带通用的 Token）。
  - 网关（如 Nginx、Kong、APISIX 或者自己用 Node.js 写的 BFF）根据 URL 前缀进行反向代理。比如匹配到 `^/api/user/(.*)$` 就 `proxy_pass http://internal-user-svc:8080/$1`。
* **动态环境变量注入 (Build Once, Run Anywhere)**：
  - 抛弃 Webpack/Vite 传统的 `process.env.VUE_APP_API` 编译时替换。因为一旦打包，它就成了写死在 `app.js` 里的字符串，无法做到一套 Docker 镜像在多个环境流转。
  - 最佳实践：在 `index.html` 的 `<head>` 里，让 Nginx 或 Node 服务器在吐出页面前，动态注入一段 `<script>window.__ENV__ = { API_URL: '...' }</script>`。前端代码运行时去读 `window.__ENV__`。

## 3. 开箱即用：核心代码骨架 (Implementation)

以在 Nginx 和前端工程化结合的场景下，如何优雅地实现这套同源网关映射和动态环境。

### 3.1 统一的同源入口（Nginx 生产环境网关配置）

前端和后端的请求在物理层面上汇聚在同一个 Nginx（或者微服务 Ingress）。前端代码中再也不需要写全称域名，直接写 `/api/xxx` 即可（同域，绝对不存在 CORS 跨域问题）。

```nginx
server {
    listen 80;
    server_name app.mycorp.com;

    # 1. 托管前端的静态资源 (React/Vue dist)
    location / {
        root /usr/share/nginx/html;
        try_files $uri $uri/ /index.html;
    }

    # 2. 核心：微服务网关映射 (同源代理)
    # 当浏览器请求 app.mycorp.com/api/user/login 时，命中这里
    location ^~ /api/user/ {
        # 转发到内网的 Java 用户微服务集群 (K8s Service Name 或内网 IP)
        proxy_pass http://user-service:8080/; 
        # 传递真实的客户端 IP 给后端，否则后端获取到的全是 Nginx 的 IP
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_headers;
        proxy_set_header Host $http_host;
    }

    # 3. 订单微服务的映射
    location ^~ /api/order/ {
        # 转发到内网的 Go 订单微服务
        proxy_pass http://order-service:9000/;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

### 3.2 动态环境变量的注入 (K8s / Docker 零构建切换)

不要再用 `npm run build:test` 和 `npm run build:prod` 打十几个包了。打一个纯净的镜像，然后在启动容器的 `entrypoint.sh` 里通过环境变量替换配置。

**1. 前端代码 (index.html)**
在 public 目录下放一个占位符配置：
```html
<head>
  <script>
    // 这个配置会在容器启动时，被 sed 或 envsubst 命令实时替换
    window.__APP_CONFIG__ = {
      API_BASE: "REPLACE_ME_API_BASE",
      DEBUG_MODE: "REPLACE_ME_DEBUG_MODE"
    };
  </script>
</head>
```

**2. 前端请求封装 (axios.ts)**
```javascript
// 不再读取编译时的 process.env，而是读取运行时的全局挂载变量
const config = window.__APP_CONFIG__ || {};
const baseURL = config.API_BASE || '/api';

const request = axios.create({
  baseURL,
  timeout: 10000,
});
```

**3. Docker 启动脚本 (entrypoint.sh)**
这是灵魂所在。运维在 K8s 部署时，只需修改 Deployment 的环境变量（比如把环境从 staging 切换为 prod），容器启动时会自动把 `index.html` 里的 `REPLACE_ME_xxx` 替换成真实的环境变量。

```bash
#!/bin/sh

# 读取容器当前的环境变量 (比如在 K8s 中注入的 API_BASE="https://api-prod.mycorp.com")
# 然后使用 sed 强行替换静态文件目录下的 index.html 中的占位符
sed -i "s|REPLACE_ME_API_BASE|${API_BASE:-'/api'}|g" /usr/share/nginx/html/index.html
sed -i "s|REPLACE_ME_DEBUG_MODE|${DEBUG_MODE:-'false'}|g" /usr/share/nginx/html/index.html

# 启动 Nginx (或者 Node.js BFF)
exec nginx -g "daemon off;"
```

## 4. 边界情况与避坑补充 (Edge Cases & Gotchas)
* **Nginx 代理时的斜杠 (/) 截断问题**：在 `proxy_pass` 时，如果 URL 末尾带了斜杠 `http://user-service:8080/`，Nginx 会把匹配到的 `location` 路径剥离掉（比如把 `/api/user/info` 转发给后端时变成了 `/info`）。如果没带斜杠 `http://user-service:8080`，则会把整个路径原封不动地传过去（变成 `/api/user/info`）。**这个极其细微的区别是新手配网关 404 找不到接口的万恶之源。必须和后端提前约定好接口规范。**
* **前端开发联调环境下的 Proxy**：上面说的是生产环境的 Nginx。在本地开发时，前端使用 Vite 或 Webpack，必须在 `vite.config.ts` 中的 `server.proxy` 完美模拟生产环境的 Nginx 配置。`target` 指向测试环境的网关，且通过 `rewrite` 规则重写路径，确保开发和生产的请求 URL 格式 100% 保持一致（绝对不能在本地写死 `http://`，必须使用局部的反向代理）。
* **微前端环境下的全局变量污染**：如果你用了 qiankun 或 wujie 微前端，多个子应用如果都在 `index.html` 里注册了 `window.__APP_CONFIG__`，由于沙箱没隔离好的话，会互相覆盖掉对方的环境配置！**解法**：每个微服务/子应用必须使用自己独特的命名空间（如 `window.__APP_CONFIG_USER__`），防止环境配置雪崩。
