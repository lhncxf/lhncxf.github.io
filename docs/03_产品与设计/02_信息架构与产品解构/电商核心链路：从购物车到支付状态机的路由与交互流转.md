# 电商核心链路：从购物车到支付状态机的路由与交互流转

在老牌前端工程师的眼里，电商的“交易漏斗”从来不是一张简单的 UI 视觉图，而是一张严密的**路由拓扑图 (Router Graph)**。从用户点击“去结算”的那一刻起，我们就进入了一个受控的、单向的、不可逆的**全局状态机**。

如果你把整个下单流程看作一个 `main.js` 入口，那么购物车、确认订单、支付网关就是不同的路由模块。每一个模块的跳转，不仅是 `pushState` 的位置变化，更是后端状态、本地缓存、以及用户心理预期的多重同步。

## 1. 路由拓扑：单向数据流的强制执行

在设计核心链路的路由时，我们必须遵循 **"One-way Navigation"** 原则。

*   **购物车 (Cart) -> 确认订单 (Checkout)**：这是一个 `router.push`。用户在这个阶段依然有较高的自由度，可以随时点击 Back 返回修改商品数量。
*   **确认订单 -> 支付 (Payment)**：这是一个关键的 `router.replace`。一旦生成订单并唤起支付插件，我们通常会销毁之前的确认页状态，防止用户通过物理返回键触发重复下单逻辑。这在工程上等同于处理 **幂等性 (Idempotency)** 的第一道防线。

### 路由守卫的逻辑映射
在这一链路中，全局路由守卫（BeforeEach）扮演了“业务校验器”的角色。比如，在进入 `checkout` 路由前，必须拦截并校验：
1.  **登录态 (Auth Token)**：没登录？重定向到 Login。
2.  **SKU 有效性**：库存是否归零？价格是否发生变动？如果校验失败，路由应当被 `next(false)` 强制拦截，并弹出 `Toast` 提示，而不是让用户看到一个空白的骨架屏。

## 2. 支付状态机：解决 UI 的“中间态”焦虑

支付环节是整个链路中复杂度最高、边界条件最密集的区域。从前端架构看，这本质上是一个**有限状态机 (FSM)**。

我们需要处理以下核心状态：
*   **PENDING (待支付)**：前端已生成订单，正在唤起微信/支付宝 SDK。此时 UI 应当进入一个全屏的 Loading 或不可交互状态，防止用户二次点击。
*   **PROCESSING (支付中)**：用户已输入密码，但后端尚未收到回调。这时候最忌讳的是让用户手动刷新。我们需要一个**轮询机制 (Polling)** 或者 **WebSocket** 来监听后端的最终判决。
*   **SUCCESS (支付成功)**：这是 Happy Path。跳转到 `success` 路由，并清除本地的购物车缓存。
*   **FAIL (支付失败)**：进入 Unhappy Path。不能简单地报错，而要引导用户进入“重新支付”或“查看订单详情”的分支逻辑。

## 3. 应对 Unhappy Paths：防御式交互设计

作为资深开发者，我们更关心“翻车”了怎么办。网络超时、用户取消、支付延迟回调，这些都是我们要处理的 `catch` 块。

### 场景 A：网络超时与支付挂起
当用户完成支付但网络环境极差，前端拿不到回调结果时，UI 会卡在 `PROCESSING` 状态。
*   **解决方案**：引入 **"Status Recovery"** 机制。在用户下次打开 App 或回到订单列表时，前端主动触发一次订单状态的主动同步。这就像是 Git 的 `fsck`，用来修复本地状态与远程仓库的不一致。

### 场景 B：支付回调延迟
后端收单成功，但通知前端的消息还没到。如果此时用户手动返回订单页，看到的状态依然是“待支付”，这会导致极大的用户恐慌（“我明明付了钱，怎么还叫我付？”）。
*   **优化策略**：在支付完成的那一刻，前端可以先设置一个**乐观状态 (Optimistic Update)**，或者在订单列表页显示一个“支付结果确认中”的中间态，而不是死板地显示数据库里的旧数据。

## 4. 信息架构：组件树的深度控制

在 `Checkout` 页面，信息架构的组织决定了代码的可维护性。
*   **父组件 (OrderProvider)**：负责维护整个订单的 `Snapshot`，包括地址 ID、优惠券 ID、商品清单。
*   **子组件 (Atomic Components)**：
    *   `AddressPicker`：纯展示与选择，通过 `emit` 修改父组件的状态。
    *   `PriceCalculator`：这是一个 **Memoized Selector**。每当优惠券或地址（涉及运费）变化时，自动重新计算总价。
    *   `PayButton`：作为最后的 `Action` 执行者，它不应该包含任何业务逻辑，只负责触发父组件定义的 `submitOrder` 方法。

## 总结

电商核心链路的构建，本质上是在不稳定的网络环境和复杂的业务逻辑中，通过**严格的路由流转**和**清晰的状态机定义**，为用户构建一个稳定的、可预测的交互闭环。

对于一个有十年经验的前端来说，我们不关注按钮的颜色是不是够蓝，我们关注的是当用户在地下铁、电梯里断网的那一秒钟，我们的代码能不能像一个老练的管家，妥善地处理掉那个悬而未决的 `Promise`。
