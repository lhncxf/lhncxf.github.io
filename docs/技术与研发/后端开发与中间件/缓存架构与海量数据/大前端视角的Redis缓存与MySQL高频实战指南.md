# 大前端视角的 Redis 缓存与 MySQL 高频实战指南

对于拥有 10 年经验的前端开发者来说，转型全栈或深入 Node.js 服务端开发时，数据库层（DB Layer）不再是黑盒。我们不需要像 DBA 那样精通调优，但必须理解数据流转的本质，尤其是如何通过缓存与数据库的配合解决高并发、数据一致性以及性能瓶颈。

## 一、 Redis 核心场景：从 KV 到复杂数据结构

在 Node.js 服务端，Redis 不只是简单的“内存 Key-Value”，它的多种数据结构是处理前端复杂交互的利器。

### 1. ZSet：排行榜与时间轴的“物理外挂”
前端在做实时排行榜（如：直播间贡献榜、积分排位）或按时间倒序的信息流时，如果直接扫 MySQL，在大数据量下会非常吃力。
*   **核心逻辑**：利用 Redis 的 `Sorted Set`。将用户 ID 作为 `member`，分数或时间戳作为 `score`。
*   **资深视角**：`ZREVRANGE` 可以在 $O(\log N + M)$ 时间内取出 Top N。相比 MySQL 的 `ORDER BY` + `LIMIT`，它能支撑极高的 QPS 且响应极其稳定。
*   **代码投影**：
    ```javascript
    // 更新用户分数：ZADD ranking 1500 user_888
    // 取出前10名：ZREVRANGE ranking 0 9 WITHSCORES
    ```

### 2. Hash 与 JSON 的选型博弈
存储用户信息或配置对象时，是直接存一个 `Stringified JSON`，还是用 `Hash` 结构？
*   **方案 A (String)**：简单直接，但如果对象很大，每次只改一个字段也得全量读写，网络 IO 浪费严重。
*   **方案 B (Hash)**：支持 `HSET/HGET` 单个字段。适合字段多、变更频繁的场景。
*   **选型心法**：如果对象结构固定且经常局部更新（如：用户配置、购物车项），首选 **Hash**；如果是作为整体读取且结构不固定，则存 **JSON String** 以保持灵活性。

## 二、 旁路缓存与双写一致性（Cache Aside Pattern）

这是全栈开发中最核心的命题：如何保证 Redis 和 MySQL 里的数据是一样的？

### 1. 为什么“先更数据库，再更缓存”是错的？
如果两个并发请求 A 和 B，A 先更了 DB，B 紧接着更了 DB。但由于网络波动，B 的缓存更新先到了，A 的后到。结果：DB 是 B 的值，缓存却是 A 的旧值。脏数据诞生。

### 2. Node.js 里的最佳实践：Cache Aside
*   **读流程**：先读 Cache -> 命中则返回 -> 未命中读 DB -> 异步回写 Cache。
*   **写流程**：**先更数据库 -> 立即删除缓存**。
*   **认知刷新**：为什么是“删”而不是“更新”？因为“删”是幂等的，且能强制下一次读请求触发延迟加载，避免并发导致的缓存与 DB 不一致。

## 三、 秒杀并发与分布式锁 (Redlock)

前端在做“连击防护”，后端则要死守“不超卖”。

### 1. 并发下的“负库存”悲剧
简单的 `if (stock > 0) { stock -= 1 }` 在高并发下会因为多个请求同时进入 `if` 块而导致库存变成负数。

### 2. 分布式锁与 Redlock 算法
在分布式（多实例）环境下，必须用 Redis 充当全局锁中心。
*   **核心指令**：利用 `SET nx ex` 实现“加锁+过期”原子化。
*   **防误删**：加锁时存入一个唯一的 `RequestID`。解锁时校验 `value` 是否匹配，防止 A 逻辑太慢锁过期了，B 拿到锁，结果 A 执行完把 B 的锁给删了。
*   **高级方案**：对于更高要求的场景，直接引用 `redlock` 库实现红锁算法，在多个 Redis 节点间达成共识，防止单点故障。

## 四、 MySQL 索引秘籍：为什么你的查询慢？

理解 MySQL 索引，本质是理解如何让数据库少做“全表扫描”。

### 1. B+ 树与覆盖索引 (Covering Index)
*   **B+ 树 (默认)**：层级低，叶子节点有序且有链表相连。适合范围查询。
*   **开发者视角**：尽量做到“索引覆盖”。如果你的索引包含字段 `name`，而你只需要 `SELECT name`，MySQL 直接在索引树上就拿到了结果，根本不需要回表（去磁盘找原始行数据），速度快一个量级。

### 2. 最左前缀原则 (Left-Prefix Rule)
如果你创建了联合索引 `(a, b, c)`：
*   **生效**：`WHERE a=1` / `WHERE a=1 AND b=2`。
*   **失效**：`WHERE b=2`（跳过了 a）。
*   **设计建议**：把**过滤性最强**（去重后数量最多）的字段放最前面。

## 五、 总结：从前端到全栈的思维跨越

1.  **数据路由意识**：不是所有数据都要存 DB，高频、临时、需要计算顺序的数据扔给 Redis。
2.  **原子化意识**：并发场景下，放弃 JS 的逻辑判断，优先使用 Redis 原子指令或 Lua 脚本。
3.  **一致性权衡**：Cache Aside 能解决 99% 的问题，极端场景（如金融）需要结合事务与 Canal 订阅日志。
4.  **性能可预测性**：写 SQL 之前，先在脑子里过一遍这行代码会扫几行数据，有没有用到索引。
