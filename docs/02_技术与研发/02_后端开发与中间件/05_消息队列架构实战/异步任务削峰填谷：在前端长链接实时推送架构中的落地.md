# 单点实战与最佳实践 [异步任务削峰填谷：在前端长链接实时推送架构中的落地]

> **使用场景**：面对瞬时极高并发（如秒杀、抢红包、海量 Excel 导出、10万人同时在线直播弹幕），前端不能用普通的同步 HTTP 请求让用户死等转圈。必须在后端引入消息队列（MQ）进行异步解耦与“削峰填谷”，同时在前端配合长链接（WebSocket / SSE / 长轮询）实现进度的实时回传，打造“提交任务 -> 异步排队 -> 实时进度通知”的闭环。

## 1. 痛点与需求场景 (Context)
* **原始痛点**：
  - **同步接口超时血崩**：前端点击“导出 100 万行报表”，后端如果是同步查库生成 Excel，至少需要 30 秒。而 Nginx 或浏览器默认的 HTTP 超时时间可能只有 15 秒或 60 秒。结果就是接口 504 Gateway Timeout，前端一直转圈然后白屏报错，数据库的 CPU 被拖垮，拖累所有其他用户的正常访问。
  - **瞬时流量冲垮数据库（没做削峰）**：周杰伦演唱会抢票，10 万人同时在一秒内点击“下单”。如果用传统的微服务直连数据库，MySQL 会瞬间被上万的并发连接打挂（Too many connections / 死锁）。
* **预期目标**：
  - **异步化解耦**：前端发起高耗时/高并发请求，后端瞬间返回 200 和一个 `taskId`（任务 ID），接口耗时降为 50ms。
  - **排队削峰**：后端将真实的业务负载扔进 Kafka 或 RabbitMQ 队列。消费端（Worker）根据自身算力（如每秒处理 100 个订单）慢慢消化，保护数据库不被打穿。
  - **前端实时进度感知**：前端拿到 `taskId` 后，建立 WebSocket 或 SSE（Server-Sent Events）通道，监听该任务的进度 `0% -> 50% -> 100%`，最后给出下载链接或支付结果。

## 2. 核心架构与设计思路 (Design & Best Practice)
* **生产者 - 消费者模型 (Producer-Consumer)**：
  - 核心思想是将“接收请求”和“处理请求”物理隔离。API 网关/BFF 层作为“生产者”，只做鉴权、生成任务 ID 和发消息到 MQ。独立的微服务集群作为“消费者”，监听 MQ 并执行重负载。
* **为什么是削峰填谷？**
  - **削峰（Peak Shaving）**：把 10 万次/秒的并发洪峰，截留在高吞吐量的消息队列里（Kafka 轻松抗住百万 TPS）。
  - **填谷（Valley Filling）**：在随后的几十秒/几分钟内，消费者以匀速（如 1000次/秒）缓慢处理掉积压的消息，充分利用服务器的闲置算力，化解瞬时崩溃。
* **前端通知协议选型**：
  - **WebSocket**：双向全双工通信。适合直播弹幕、IM 聊天、协同文档（如 Figma）。缺点是协议重、代理层（Nginx）需要特殊配置（如长链接保活、防止断开）。
  - **SSE (Server-Sent Events)**：单向（服务器 -> 客户端）通信。基于标准 HTTP 协议，对反向代理和防火墙极其友好，且自带断线重连。非常适合“异步任务进度推送”、“大屏数据实时刷新”、“ChatGPT 流式打字机输出”等场景。本篇重点推荐！

## 3. 开箱即用：核心代码骨架 (Implementation)

以 Node.js BFF 层为例，展示如何结合 Redis 消息队列（BullMQ 或 RabbitMQ 的平替）与前台 SSE，实现优雅的异步任务处理闭环。

### 3.1 BFF 层：接收前端请求并压入队列 (Producer)

当用户点击“生成海量报表”，BFF 只花 10ms 生成一个唯一 ID 并丢给队列。

```javascript
// Express.js 路由层 (生产者)
const express = require('express');
const { Queue } = require('bullmq'); // 基于 Redis 的高性能 Node.js MQ
const { v4: uuidv4 } = require('uuid');

const reportQueue = new Queue('ReportExport', {
  connection: { host: '127.0.0.1', port: 6379 }
});

const app = express();

app.post('/api/export-report', async (req, res) => {
  // 1. 鉴权与校验参数
  const { dateRange, deptId } = req.body;
  const taskId = `task_${uuidv4()}`;

  // 2. 将耗时任务推入 MQ 队列（立刻削峰）
  await reportQueue.add(
    'generateExcel', 
    { dateRange, deptId, taskId },
    { jobId: taskId, removeOnComplete: true } // 可配置防重防丢失
  );

  // 3. 瞬间响应前端（不再同步等待）
  return res.status(202).json({
    code: 0,
    msg: '任务已提交，请监听进度',
    data: { taskId } // 告诉前端，请拿这个凭证去听进度
  });
});
```

### 3.2 Worker 层：排队消费任务并更新进度 (Consumer)

部署在后台的专门算力机器上，不受 API QPS 的影响。

```javascript
// Worker.js (消费者)
const { Worker } = require('bullmq');
const redisClient = require('./redisConfig'); // 共享的 Redis，用于存储进度状态

const worker = new Worker('ReportExport', async (job) => {
  const { dateRange, deptId, taskId } = job.data;
  console.log(`[Worker] 开始处理极其耗时的任务: ${taskId}`);

  try {
    // 模拟 10 万行数据库查询与 Excel 构建（耗时 20 秒）
    for (let progress = 0; progress <= 100; progress += 20) {
      await new Promise(r => setTimeout(r, 2000)); // 模拟每一批次的计算时间
      
      // 【核心】：将实时进度写入 Redis（供前端 SSE 通道去捞）
      // 或直接通过 Socket.io 广播给特定的 userId
      await redisClient.setex(`progress:${taskId}`, 3600, progress); // 存活 1 小时
      
      // 也可以更新 job 的内部进度 (如果用 BullMQ 自带的)
      await job.updateProgress(progress);
      console.log(`任务 ${taskId} 进度: ${progress}%`);
    }

    // 任务完成，生成最终的下载 URL
    const fileUrl = `https://oss.mycorp.com/reports/${taskId}.xlsx`;
    await redisClient.setex(`result:${taskId}`, 3600, JSON.stringify({ status: 'done', url: fileUrl }));
    
    return { fileUrl }; // 返回给 MQ 引擎标记完成
  } catch (err) {
    await redisClient.setex(`result:${taskId}`, 3600, JSON.stringify({ status: 'error', error: err.message }));
    throw err;
  }
}, {
  connection: { host: '127.0.0.1', port: 6379 },
  concurrency: 5 // 【削峰限流核心】：这台机器最多同时处理 5 个 Excel，即使瞬间来了 10 万个请求，也是 5 个 5 个慢慢消化。
});
```

### 3.3 BFF 层：向前端暴露 SSE 实时进度通道

前端拿到 `taskId` 后，立刻建立单向的流式通道。

```javascript
// Express.js SSE 路由
app.get('/api/task-progress/:taskId', async (req, res) => {
  const { taskId } = req.params;

  // 1. 设置 SSE 特有的 HTTP 响应头
  res.writeHead(200, {
    'Content-Type': 'text/event-stream', // 声明这是事件流
    'Cache-Control': 'no-cache', // 禁止浏览器或代理缓存
    'Connection': 'keep-alive', // 保持 TCP 长链接不中断
  });

  // 告诉前端通道已建立
  res.write(`data: ${JSON.stringify({ status: 'connected', progress: 0 })}\n\n`);

  // 2. 定时轮询 Redis（或使用 Redis Pub/Sub），向前端推送最新进度
  const timer = setInterval(async () => {
    // 从共享缓存中读取进度或结果
    const progress = await redisClient.get(`progress:${taskId}`);
    const resultStr = await redisClient.get(`result:${taskId}`);

    if (resultStr) {
      // 任务彻底完成或报错了
      const result = JSON.parse(resultStr);
      // 推送最后一条消息，前端收到后可以拿到 URL 并断开连接
      res.write(`data: ${JSON.stringify(result)}\n\n`);
      clearInterval(timer);
      res.end(); // 关闭通道释放服务器资源
    } else if (progress) {
      // 任务进行中，推送进度给前台画进度条
      res.write(`data: ${JSON.stringify({ status: 'processing', progress: Number(progress) })}\n\n`);
    }
  }, 1000); // 每秒推一次

  // 3. 客户端主动断开（比如用户关了页面），服务器清理定时器防内存泄露
  req.on('close', () => {
    clearInterval(timer);
  });
});
```

### 3.4 前端极简监听代码 (浏览器原生支持)

```javascript
// FrontEnd.js
const startExport = async () => {
  // 1. 发起普通 POST 请求提交任务（瞬间返回）
  const { data: { taskId } } = await axios.post('/api/export-report', { /*...参数*/ });

  // 2. 原生 EventSource 开启 SSE 长链接监听
  const eventSource = new EventSource(`/api/task-progress/${taskId}`);

  // 3. 监听服务器单向推过来的消息
  eventSource.onmessage = (event) => {
    // 这里的 event.data 就是后端 res.write("data: {...}\n\n") 里的 {...}
    const payload = JSON.parse(event.data);
    
    if (payload.status === 'processing') {
      console.log(`生成进度: ${payload.progress}%`);
      // 更新 UI 进度条...
    } else if (payload.status === 'done') {
      console.log('生成完毕，下载地址:', payload.url);
      window.open(payload.url); // 自动下载
      eventSource.close(); // 前端主动断开通道，释放资源
    } else if (payload.status === 'error') {
      console.error('生成失败:', payload.error);
      eventSource.close();
    }
  };

  eventSource.onerror = (err) => {
    console.error('SSE 通道发生异常断开（浏览器会自动重连）', err);
  };
};
```

## 4. 边界情况与避坑补充 (Edge Cases & Gotchas)
* **网关层的长连接截断陷阱**：当你把这套系统部署到生产环境的 Nginx / Kubernetes Ingress 后，你会发现 SSE 几分钟就被强行挂断了，或者根本收不到流式数据而是“攒到最后一起返回”。**解法**：必须在 Nginx 配置中针对这个 SSE 路由关掉缓冲，并延长超时。`proxy_buffering off;` `proxy_read_timeout 86400s;`。
* **消息队列的幂等性消费 (Idempotent Consumer)**：因为网络重传机制（如 TCP 抖动导致 ACK 丢失），MQ 引擎可能会把同一条消息（同一个任务）重复投递给 Worker 两遍（At Least Once 语义）。如果在交易扣款场景，这会导致用户被扣两次钱。**必须在 Worker 代码的第一行，利用数据库唯一索引（Unique Key）或 Redis `SETNX` 做防重校验，确保一个 `taskId` 绝对不会被执行两次（幂等）。**
* **前端的长链接幽灵重连（幽灵并发）**：浏览器的 `EventSource` 有个特性：如果在连接没主动 `close()` 的情况下服务器断开（比如后端发版重启了），浏览器会自动、无限次地疯狂尝试重连！如果你在单页应用（SPA）中切换了路由但忘记调用 `eventSource.close()`，旧页面的通道会在后台幽灵般重连，导致成倍消耗后端并发数。**解法**：在 React `useEffect` 的 `return () => cleanup` 或 Vue 的 `onUnmounted` 生命周期中，必须强制调用 `.close()`。
