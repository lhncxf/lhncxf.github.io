# 架构溯源与深度认知：微前端 (qiankun & wujie) 选型模型与治理拆解

> **10年前端老炮的认知沉淀**：微前端从不是提升性能的“银弹”，而是一种解决大型组织协同痛点的“架构妥协”。在面对几十人协作的复杂 B 端场景或难以撼动的巨石遗留系统时，微前端的价值才真正显现。本文通过对 qiankun 与 wujie 源码及其底层原理的深度调研，拆解其沙箱隔离、样式治理及通信机制的本质。

## 1. 架构选型深度：qiankun 与 wujie 的底层博弈

在技术调研中，单纯对比 GitHub Stars 毫无意义，必须看其对浏览器原生特性的利用方式。

*   **qiankun (基于 single-spa + Proxy 沙箱)**：
    *   **原理探究**：它通过劫持 `hashchange` 或 `popstate` 事件实现路由分发，并利用 `Proxy` 代理 `window` 对象实现 JS 沙箱。
    *   **老炮视角**：其强项在于生态极度成熟，生命周期控制精准。但 qiankun 的样式隔离（尤其是 Shadow DOM 模式）存在天然硬伤，会导致三方 UI 组件库（如 AntD/Element）的弹窗定位因找不到 `body` 而彻底失效。
*   **wujie (基于 WebComponent + iframe)**：
    *   **原理探究**：wujie 巧妙地将 JS 运行环境放在 iframe 中（保证了最彻底的 JS 隔离），同时将 DOM 渲染在 WebComponent 的 ShadowRoot 中。
    *   **认知拆解**：这种“容器化”设计解决了传统 iframe 无法撑开高度、弹窗无法全屏的致命缺陷。对于追求极致隔离、不想在 JS 沙箱逃逸上浪费时间的场景，这是一种更高维度的设计思路。

**调研结论**：存量系统改造、需要精细化生命周期钩子的，选 `qiankun`；追求极致隔离、希望子应用像普通组件一样嵌入且不影响主应用环境的，选 `wujie`。

## 2. 核心痛点治理：沙箱逃逸与副作用监控

### 2.1 qiankun 的“半吊子”沙箱与异步坑点
通过阅读 qiankun 源码发现，其 `ProxySandbox` 主要拦截的是同步的全局变量读写。对于 `setTimeout`、`Promise` 或全局 `EventBus` 产生的异步副作用，如果子应用未在 `unmount` 时手动清理，极易导致基座内存泄漏。

**认知重塑：防御性拦截器设计**
在架构设计层面，不能完全信任子应用的行为。更稳妥的做法是在基座（Main App）层面建立一套“副作用自动回收”模型，通过劫持原生事件 API，自动收集并销毁未被卸载的副作用。

### 2.2 样式隔离：从 Shadow DOM 到工程化 Namespace
qiankun 提供的 `strictStyleIsolation` 虽好，但在实际工程中几乎无法大规模应用。

**更优路径：逻辑隔离优于物理隔离**
通过 PostCSS 等构建工具，在编译阶段为子应用的所有 CSS 强制增加 `[data-micro-app="xxx"]` 属性选择器，并配合 runtime publicPath 修改。这种方案虽然在视觉上不如 Shadow DOM 优雅，但在兼容第三方复杂组件库时，稳定性是最高的。

## 3. 通信机制与弹窗治理的底层逻辑

wujie 方案中，子应用运行在 iframe 里，这意味着传统的 `window.parent` 通信会产生严重的耦合。

*   **解耦通信**：调研发现，利用微前端框架自带的 `bus`（发布订阅模型）是跨容器通信的最优解。应严禁子应用直接操作主应用的 DOM，否则微前端的隔离性将形同虚设。
*   **Shadow Boundary 突破**：当在 wujie 中遇到弹窗错位时，本质上是 `append` 操作被拦截后的重定向问题。通过配置插件拦截 `appendChild` 等 DOM API，可以精准地将弹窗导向正确的 Shadow DOM 容器。

## 4. 公共依赖：模块联邦 (Module Federation) 的认知重构

微前端最让人头疼的就是依赖重复加载。每个子应用带一个 Echarts 会导致用户内存迅速枯竭。

**认知升级**：不再推荐使用老旧的 CDN `externals` 方案，版本冲突几乎不可避免。
*   **方案对标**：Webpack 5 的 **Module Federation** 才是真正的终局方案。
*   **底层认知**：它允许应用在运行时按需拉取远程模块，实现了真正的动态依赖共享。主应用作为 Host 共享核心框架，子应用作为 Remote 按需消费。

## 5. 防腐设计与监控兜底

*   **契约化设计**：基座与子应用之间的通信必须通过 TypeScript Interface 强制约束，避免协议层面的“黑箱操作”。
*   **监控体系**：利用 `window.onerror` 捕获沙箱内的崩溃。当子应用加载失败超过阈值时，应具备自动降级至“传统 iframe”或报错引导的鲁棒性。

**总结**：微前端的本质不是拆分，而是**集成**。集成者的认知高度，决定了这套架构是团队的加速器，还是未来几年的沉重技术债。
