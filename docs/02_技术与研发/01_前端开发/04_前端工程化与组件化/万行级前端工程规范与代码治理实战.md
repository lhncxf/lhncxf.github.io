# 万行级前端工程规范与代码治理实战

在前端项目规模突破万行甚至达到十万行级别时，单纯的“代码风格一致”已不足以支撑工程的稳定性。此时的工程化核心在于**防腐（Anti-Corruption）**与**确定性（Determinism）**。

## 1. Git 工作流：从 Trunk-based 到治理闭环

对于大规模团队，分支管理不只是协作，更是发布质量的阀门。

*   **Trunk-based Development (TBD) 的进阶应用**：建议采用极简的长期分支（main），通过 Feature Flags 控制功能上线。减少长周期分支带来的合并地狱。
*   **原子化提交 (Atomic Commits)**：每个 Commit 必须是逻辑闭环的，禁止将重构与新功能混在同一个提交中。
*   **Merge Request 强制准入制**：
    *   必须通过 CI 静态扫描与单元测试。
    *   强制两人以上 Approve。
    *   利用 `CODEOWNERS` 文件锁定核心架构代码，非架构组人员修改必须获得专项授权。

## 2. 静态检查生态：工业级防腐手段

万行代码下，ESLint 不只是纠正空格，而是作为架构约束工具。

*   **ESLint 层级化治理**：
    *   基础规则（Recommended）：处理基本语法。
    *   架构约束（Project Specific）：使用 `eslint-plugin-import` 限制跨模块调用。例如，禁止 `features` 模块直接引用 `pages` 下的内容。
    *   TypeScript 增强：强制开启 `no-explicit-any` 与 `explicit-module-boundary-types`。
*   **Stylelint 与 CSS 变量约束**：禁止在业务代码中硬编码颜色值（Hex/RGB），强制使用设计系统定义的变量。
*   **全量 vs 增量扫描**：
    *   本地阶段（Husky + lint-staged）：仅针对增量文件，保证提交效率。
    *   CI 阶段：执行全量扫描，防止因依赖升级或配置变更导致的存量问题。

## 3. Git Hooks 与提交规范：数据资产的基石

日志记录了项目的演进脉络，规范的日志意味着可追溯性。

*   **Commitlint 强约束**：遵循 Angular 规范，但应结合业务增加 `scope` 必填项，如 `feat(order): add refund logic`。
*   **Husky 的防绕过机制**：利用 `pre-commit` 拦截格式问题，`commit-msg` 拦截语义问题。对于大型团队，必须在服务器端（GitLab/GitHub Action）再次校验，防止开发者使用 `--no-verify` 跳过本地检查。

## 4. Monorepo 边界与依赖治理

当项目演进为 Monorepo 时，最大的挑战是依赖膨胀与循环引用。

*   **Package 边界定义**：
    *   `apps/*`: 零逻辑，仅负责聚合模块与环境变量注入。
    *   `features/*`: 独立业务域，模块间禁止水平交叉引用。
    *   `shared/*`: 纯工具函数、Hooks。
    *   `ui/*`: 领域无关的 UI 组件库。
*   **依赖分层准则**：底层 package 禁止依赖上层 package。利用 `pnpm-workspace.yaml` 严格控制可见性。
*   **幽灵依赖与依赖项锁定**：强制使用 `pnpm`，利用其硬链接机制解决 `node_modules` 拓扑问题，配合 `changeset` 进行多包发布版本管理。

## 5. 自动化 CI/CD：全链路确定性

CI 不是脚本的堆砌，而是质量控制的流水线。

*   **流水线阶段优化**：
    *   **Phase 1: Verify** (Lint + TypeCheck + Unit Tests) - 快速反馈。
    *   **Phase 2: Build** (Production Build + Bundle Analysis) - 检查体积变化。
    *   **Phase 3: Integration** (E2E Test) - 模拟真实链路。
*   **产物分析报告**：CI 自动对比构建前后的 `stats.json`，若包体积增长超过 5%，自动触发 Warning 或 Block。

## 6. 技术债治理：长效演进方案

治理万行工程是一场持久战。

*   **TODO 指数监控**：利用脚本统计全量代码中的 `TODO`、`FIXME` 数量及停留时长，纳入团队技术健康度看板。
*   **弃用标记 (Deprecated)**：对于旧组件，使用 `@deprecated` 注释并配合 ESLint 报错，逐步引导开发者迁移。
*   **模块化重构律**：执行“童子军军规”——离开时代码比进来时更干净。每次功能迭代强制包含 10% 的存量清理。

## 结语

前端工程化不是单纯的配置。它是通过强制性的工具链和科学的流程，将开发者的自由度限制在“产生高质量代码”的范围内。在万行级的战场上，流程优于个人英雄主义。
