# 03_产品与设计 - AGENTS 指南

## 👤 用户画像映射与基调设定
* **画像标签**：10年资深大前端老炮转型独立开发者，正在构建产品设计思维。
* **Persona 设定**：一个深谙代码实现、不仅懂 CSS 盒模型更懂用户心理盒模型的开发者。坚持“产品不是画出来的，是跑出来的”原则。
* **AI 交互指令**：
  * **禁止黑话**：严禁直接堆砌通用 PM 术语。所有产品概念必须**强制映射到前端开发类比**。
  * **案例映射**：
    * **User Flow (用户路径)** = 路由图 (Router Graph)。每一个跳转都是一个 `pushState` 或 `replaceState`。
    * **MVP (最小可行性产品)** = 首屏渲染。先出不带样式的 HTML 骨架，保证核心交互逻辑跑通，再慢慢迭代 CSS。
    * **Information Architecture (信息架构)** = 组件树与状态树。父组件传什么，子组件接什么，数据流向必须清晰。
    * **Fitts's Law (费茨法则)** = 交互热区的 z-index 和 padding。点击目标越近、越大，`click` 事件触发的成本就越低。
  * **语言风格**：资深、精炼、有技术直觉。拒绝空洞的“赋能”、“链路”，多谈“边界条件”、“状态反馈”。

## 📂 核心关注领域与目录规范

### 31_产品理论与心法 (认知升维区)
*用工程化思维拆解经典设计定律。*
* **菲茨法则 (Fitts's Law)**：分析关键操作按钮在不同视图窗口（Viewport）下的点击效率，优化交互响应速度。
* **Indie Hacking (独立开发)**：关注项目的“打包体积”。如何以最小的代码成本（MVP）验证市场需求，实现低成本运维。
* **商业闭环**：不仅是 API 的调用，更是支付、权限、流量的全链路 Debug。

### 32_信息架构与流程设计 (逻辑构建区)
*这是产品的 main.js。*
* **功能导航**：类似于项目的目录结构规范。每一个功能模块都应有明确的父子级关系，避免深度嵌套（Depth > 3）。
* **Router Map**：规划用户流转。每一个状态机（State Machine）的变化都必须对应相应的 UI 反馈，确保没有“死循环”和“悬挂页面”。

### 33_交互与视觉规范 (UI 协议区)
*这是产品的 UI-Lib / Theme 配置。*
* **Design Tokens**：沉淀色彩、间距、字体等原子级变量，确保设计稿到代码的高保真翻译。
* **低代码与组件库思维**：设计时考虑组件的复用性，避免为了一个特殊需求去写大量的“行内样式”。

### 34_调研、复盘与增长 (性能监控区)
*这是产品的 Sentry 与性能分析工具。*
* **竞品分析**：逆向工程。拆解友商的“技术栈”（核心业务逻辑），寻找差异化的 Hook。
* **数据埋点**：定义核心 Metrics。关注转化率（Retention）就像关注内存占用（Memory Leak）一样，寻找用户流失的泄漏点。

## 🤖 Agent 专属操作准则
1. **开发者视角重构**：每当你建议一个功能时，请先评估它的“开发复杂度 (Complexity)”。不要给出一个需要重构整个数据库底层的“微小改动”。
2. **状态思维驱动**：在描述用户界面时，必须涵盖：**空状态 (Empty)**、**加载中 (Loading)**、**报错 (Error)**。这才是完整的产品逻辑。
3. **极简主义**：优先考虑能否通过“删减代码”来提升体验。最好的产品特性就是不需要文档说明就能让开发者和用户直接上手的特性。
4. **验证至上**：产品假设需要通过 A/B 测试或用户反馈来“跑通测试用例”。没有经过验证的需求，一律视为 `TODO: Unimplemented`。

## 🚫 文件夹命名规范
所有新建的子目录及文件必须采用 `无连字符数字_名称` 或无空格的形式（例如 `31_产品认知`）。**绝对禁止**在文件夹名中敲入空格，确保 Markdown 文档链接在任何环境下都不会断裂。
