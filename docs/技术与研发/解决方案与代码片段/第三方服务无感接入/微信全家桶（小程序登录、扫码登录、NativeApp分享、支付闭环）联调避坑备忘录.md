# 💊 疑难踩坑与排错备忘：微信全家桶（小程序登录、扫码登录、NativeApp分享、支付闭环）联调避坑备忘录

> **使用场景**：当公司业务涵盖了 PC 网站、微信公众号（H5）、微信小程序、iOS/Android 原生 App 时，前端往往需要作为“总线”，对接极其庞杂的微信开放平台与公众平台生态。本文记录了在全端接入微信体系时，前端在鉴权（OAuth2.0）、扫码登录、分享及支付链路中最致命的坑点与统一解法。

## 📌 一、 业务背景与技术痛点 (Context & Pain Points)
*“为什么微信的文档每个平台都不一样？为什么我的 `wx.config` 总是 `invalid signature`？”*
* **历史包袱（碎片化的微信生态）**：很多前端新手接到的第一个任务是“做个微信分享”。点开微信文档一看：微信公众平台（管公众号和H5）、微信开放平台（管网站扫码和 App）、微信商户平台（管收钱的）、微信小程序后台。四个平台的 AppID 和 AppSecret 全不一样，且完全隔离。
* **开发协同灾难（沙盒与真机的鸿沟）**：在本地 `localhost` 调试微信支付和授权登录，由于微信严格的白名单回调域名限制（甚至必须是备案过的 HTTPS 域名），本地根本跑不通。一放到线上真机，稍微带个特殊的标点符号参数，签名就直接失败。更别提由于安卓和 iOS 的微信 Webview 内核不同，导致的各种诡异的返回刷新问题。
* **选型诉求**：我们需要一套大一统的“微信网关层（BFF）”和前端标准的“抹平差异的 JS-SDK 封装层”。不让业务代码里充斥着 `if (isWechat) ... else if (isApp) ...`。

## 💡 二、 核心机制解构 (Mental Model)
*万变不离其宗：微信的所有鉴权，本质都是 OAuth 2.0 的“拿 Code 换 Token”。*

### 2.1 微信公众号（H5）网页授权登录：静默授权 vs 手动授权
*   **底层逻辑 (OAuth 2.0)**：用户点一个链接，跳到微信服务器的特殊 URL（带上我们公司的 `appid` 和一个 `redirect_uri`）。微信一看，哦，是要授权。用户点同意后，微信把网页重定向回我们的 `redirect_uri`，并在 URL 屁股后面挂个 `?code=xxxx`。前端拿到这个极其短命的 `code`，发给后端，后端用 `code` 去微信服务器换取该用户的 `openid` 和 `access_token`（甚至拿到头像昵称）。
*   **踩坑（无限循环重定向）**：很多前端喜欢把去微信授权的动作写在 `router.beforeEach` 里。结果用户拒绝授权，跳回来没有 `code`，前端又把他踢去微信，微信又踢回来……死循环。
*   **最佳实践**：`snsapi_base`（静默授权，只能拿 `openid`，不弹窗）和 `snsapi_userinfo`（需用户同意，拿详细信息）。永远是后端把生成好的微信授权 URL 通过接口给前端，前端直接 `window.location.href` 跳转。不要前端自己拼装那一大长串乱七八糟的 URL。

### 2.2 PC 网站微信扫码登录：二维码其实是个 Iframe
*   **痛点**：PC 端让用户掏出手机扫码登录。
*   **解法**：去“微信开放平台”（注意不是公众平台）申请一个网站应用。前端页面引入微信的 `wxLogin.js`。在页面上 `new WxLogin({...})`。它的本质是往你的 `div` 里塞了一个微信官方域名的 `iframe`。用户用手机扫那个 `iframe` 里的二维码并确认后，那个 `iframe` 会自己重定向到你配置的 `redirect_uri`，并带上 `code`。剩下的流程和 H5 一模一样。
*   **高阶体验（内嵌扫码样式定制）**：嫌微信默认的二维码太丑？你可以通过传 `href` 参数，把你放在自己 CDN 上的一个极其变态的 CSS 文件的绝对路径丢给微信，微信会在 `iframe` 里加载你的 CSS，从而把你公司的 Logo 盖在二维码上，或者把字体变透明。

### 2.3 微信 JS-SDK（分享、扫一扫）：致命的 `invalid signature`
要在 H5 网页里调用微信的扫一扫、隐藏右上角菜单、或者自定义分享（把标题和缩略图换掉）。
1.  前端把当前页面的**完整 URL（包含参数，去掉 `#` 后面的 hash）**发给后端。
2.  后端用自己的 `jsapi_ticket` 加上这个 URL，按照 ASCII 字典序算出一个 `signature` 给前端。
3.  前端调用 `wx.config({ appId, timestamp, nonceStr, signature, jsApiList })`。
**避坑大礼包**：
*   **坑 1（URL 编码问题）**：后端算的 URL 和前端给的 URL 差了一个 `/`，或者前端 URL 里的参数被 `encodeURIComponent` 转义了后端没解开，必报 `invalid signature`。
*   **坑 2（iOS 与 Android 的爱恨情仇）**：在 Vue/React 这种单页应用（SPA）里，由于是 `history.pushState` 路由。iOS 微信极度死板，它只认用户**第一次进入网页时**的那个根 URL 来做签名校验。而 Android 每次切换路由，URL 变了，它就要求重新拿新 URL 签名。
*   **终极解法**：在全局路由守卫里，判断 `isIOS`，如果是，把第一次进入的 URL 存进 Pinia 全局变量。只要是 iOS 去调 `wx.config`，永远死死咬住那个存下来的初始 URL 发给后端要签名；Android 正常传 `window.location.href`。

## ⚙️ 三、 支付闭环：防抖、异步轮询与 Webhook 兜底
*微信支付（JSAPI 支付 / App 支付 / 小程序支付）*

*   **流程**：前端告诉后端要买什么 -> 后端去微信商户平台下单拿回预支付交易会话标识（`prepay_id`）并再次签名 -> 把这些杂七杂八的（`appId`, `timeStamp`, `nonceStr`, `package`, `signType`, `paySign`）给前端 -> 前端调对应的 `wx.requestPayment`。
*   **极其致命的坑（用户切后台）**：用户在微信里输完密码，此时还没点“完成”，他就直接切到了微信聊天列表，或者强杀了微信进程。那前端在 `success` 回调里写的跳转逻辑永远不会执行，用户会大骂钱扣了没发货。
*   **兜底闭环**：绝对不能相信前端的 `success` 回调。前端发起支付后，立刻拉起一个全屏的 `Loading` 遮罩，并开启一个定时器（`setInterval`），每 2 秒查一次后端的订单状态接口（或者建立 WebSocket 长连接）。如果后端收到了微信服务器的异步 Webhook 通知，订单变成已支付，后端的查询接口才返回 `true`。前端这个时候再关掉 Loading，跳转到支付成功页。这才是企业级的支付防雪崩闭环。

## 📝 四、 沉淀与复盘 (Takeaways)
*   **认知反转**：不要和微信文档较劲，它的历史包袱比我们还重。把所有和微信打交道的加解密、拼参数、算签名（无论是 MD5 还是 RSA），全部推给后端（BFF 层）。前端拿到的，永远是一个开箱即用的干净 JSON 或者直接可以 `href` 跳的 URL。前端要做的是极端的兼容性兜底（抹平 iOS/安卓的坑、抹平 App/小程序的 API 差异）。
